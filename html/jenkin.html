1Ô∏è‚É£ What is Jenkins?

Think of Jenkins as a DevOps robot that builds, tests, and deploys your app whenever code changes.

Jenkins is an open-source automation server used mainly for CI/CD (Continuous Integration and Continuous Delivery/Deployment). It automates the process of building, testing, and deploying applications.

Jenkins in simple words

Instead of you manually doing:

  • git pull
  • mvn clean install / npm install
  • pytest / JUnit
  • docker build / scp / kubectl apply

...you let Jenkins do all of this automatically on every commit or pull request.

How Jenkins behaves (mental model)

  • Watches your Git repo (GitHub, GitLab, Bitbucket, etc.).
  • On every push / PR:
    • Pulls latest code
    • Builds it
    • Runs tests
    • Packages the app
    • Deploys (if configured)

4Ô∏è‚É£ Jenkins Core Concepts

These are the concepts interviewers expect you to know clearly.

4.1 Jenkins Server (Controller)

  • Main Jenkins instance.
  • Hosts the Web UI, job configuration, and scheduling.
  • Older name: master; now called controller.

4.2 Agents / Nodes

  • Agents (nodes) are machines where builds actually run.
  • Controller assigns jobs to agents.
  • Benefits:
    • Scale (many builds in parallel).
    • Different environments: one agent with Java, one with Docker, one with Android SDK, etc.

4.3 Jobs / Projects

A job (or project) is one unit of work Jenkins does.

  • Freestyle job: GUI-based, good for simple builds/scripts.
  • Pipeline job: uses Jenkinsfile, recommended for modern CI/CD.
  • Others: Multi-branch pipeline, folders, etc.

4.4 Jenkinsfile

Text file in your repo that defines the pipeline as code.

pipeline {
  agent any

  stages {
    stage('Build') {
      steps {
        echo 'Building...'
      }
    }

    stage('Test') {
      steps {
        echo 'Running tests...'
      }
    }

    stage('Deploy') {
      steps {
        echo 'Deploying...'
      }
    }
  }
}

Jenkins reads this file, visualizes stages, and runs them in sequence.

4.5 Declarative vs Scripted Pipelines

Declarative:

  • Newer, structured, easier syntax.
  • Uses pipeline { ... }.
  • Recommended for most use cases.

Scripted:

  • Pure Groovy code (more flexible).
  • Uses node { ... } style.
  • Better for complex logic.
# Basic scripted example
node {
  stage('Build') {
    echo 'Building...'
  }
  stage('Test') {
    echo 'Testing...'
  }
}

4.6 Triggers

How builds start automatically:

  • Webhooks: GitHub/GitLab notify Jenkins on push/PR.
  • SCM Polling: Jenkins polls repo every X minutes.
  • Cron schedule: e.g., nightly at 1 AM.
  • Manual: click ‚ÄúBuild Now‚Äù.
  • Upstream/Downstream: one job triggers another.

4.7 Plugins

Plugins extend Jenkins functionality:

  • Source control: Git, GitHub, GitLab.
  • Build tools: Maven, Gradle, Node, Python.
  • Containers: Docker, Kubernetes.
  • Notifications: Email, Slack.
  • Security & UI: Credentials, Role-based auth, Blue Ocean.

5Ô∏è‚É£ Jenkins Interview Questions & Answers

Use these as your quick revision set before a DevOps round.

Jenkins is an open-source automation server used primarily for CI/CD. It automates building, testing, and deploying software by integrating with source control systems and other DevOps tools.
  • Avoids manual builds and deployments.
  • Reduces human error in repetitive tasks.
  • Speeds up feedback to developers.
  • Creates repeatable, automated CI/CD pipelines.
A Jenkins pipeline is a set of automated steps (build, test, deploy, etc.) defined as code, usually in a Jenkinsfile, that Jenkins executes in sequence or parallel to deliver software.
A Jenkinsfile is a text file stored in the project repository that defines the Jenkins pipeline as code. It contains stages, steps, and configuration using declarative or scripted syntax.
  • Declarative pipeline:
    • Higher-level, opinionated syntax.
    • Uses pipeline { ... } block.
    • Easier to read and maintain.
    • Recommended for most use cases.
  • Scripted pipeline:
    • Based on Groovy scripting.
    • Uses node { ... } blocks.
    • More flexible; supports complex logic.
    • Slightly harder to maintain.
A Jenkins agent (node) is a machine that the Jenkins controller uses to run jobs. Agents allow Jenkins to distribute workloads across multiple systems, scale horizontally, and support different environments (different OS, tools, SDKs).
High-level steps:
  • Ensure Git plugin is installed (usually pre-installed).
  • Configure Git in Manage Jenkins ‚Üí Global Tool Configuration.
  • In job configuration, select Source Code Management: Git.
  • Add repository URL and credentials (if private).
  • Optionally add webhooks in GitHub/GitLab to trigger builds automatically.
Two common methods:
  • Webhooks (recommended):
    • Configure a webhook in Git hosting (GitHub/GitLab/Bitbucket) pointing to Jenkins URL (e.g., /github-webhook/).
    • On push/PR, Git sends event to Jenkins ‚Üí Jenkins triggers build.
  • SCM Polling:
    • Enable ‚ÄúPoll SCM‚Äù in job configuration.
    • Set cron schedule (e.g., H/5 * * * * for every 5 minutes).
    • Jenkins periodically checks for changes and builds if repo changed.
Plugins extend Jenkins functionality to integrate with external tools such as Git, Maven, Docker, Kubernetes, Slack, SonarQube, etc. They are important because they make Jenkins extremely flexible and allow it to fit into almost any CI/CD ecosystem.
  • Freestyle job:
    • Configured via Web UI.
    • Good for simple or legacy builds.
    • Logic is not stored as code ‚Üí harder to version-control.
  • Pipeline job:
    • Defined as code (Jenkinsfile).
    • Better for complex CI/CD workflows.
    • Pipeline definition is version-controlled with the application code.
  • Store credentials under Manage Jenkins ‚Üí Credentials.
  • Supported types: username/password, SSH keys, secret text, etc.
  • In pipelines, reference credentials via credentialsId and withCredentials:
    withCredentials([usernamePassword(credentialsId: 'my-creds', usernameVariable: 'USER', passwordVariable: 'PASS')]) {
      sh "echo Using user $USER"
    }
  • Never hard-code secrets directly in scripts or Jenkinsfile.
Example:
pipeline {
  agent any

  stages {
    stage('Build') {
      steps {
        echo 'Building app...'
      }
    }

    stage('Test') {
      steps {
        echo 'Running tests...'
      }
    }
  }
}
  • stages: pipeline phases.
  • stage: logical step (build, test, deploy, etc.).
  • steps: actual commands/actions inside the stage.
Example:
stage('Parallel Tests') {
  parallel {
    stage('Unit Tests') {
      steps {
        echo 'Running unit tests'
      }
    }

    stage('Integration Tests') {
      steps {
        echo 'Running integration tests'
      }
    }
  }
}
This allows independent tasks to run concurrently, speeding up the pipeline.
  • In a pipeline:
    post {
      success {
        archiveArtifacts artifacts: 'build/**', fingerprint: true
      }
    }
  • In freestyle jobs: use ‚ÄúArchive the artifacts‚Äù under Post-build Actions.
Blue Ocean is a modern UI for Jenkins (provided via plugin) that improves visualization of pipelines, stages, and logs. It offers a more user-friendly way to view and interact with Jenkins pipelines.
Short answer:
  • Run Jenkins inside Docker.
  • Use Docker on agents to build and run images.
  • Use Docker-related pipeline steps or shell commands.
Example:
stage('Build Docker Image') {
  steps {
    sh 'docker build -t myapp:latest .'
    sh 'docker push myrepo/myapp:latest'
  }
}
(Requires Docker available and configured on the agent.)

6Ô∏è‚É£ Hands-On Jenkins Examples

Conceptual steps you can actually try when you have Jenkins installed.

Example 1: Simple Freestyle Job

Goal: Create a job that prints ‚ÄúHello from Jenkins‚Äù.

  • Open Jenkins UI.
  • Click New Item.
  • Name: hello-job ‚Üí choose Freestyle project ‚Üí OK.
  • Go to Build section:
    • Linux agent: ‚ÄúExecute shell‚Äù ‚Üí echo "Hello from Jenkins"
    • Windows agent: ‚ÄúExecute Windows batch command‚Äù ‚Üí echo Hello from Jenkins
  • Save ‚Üí click Build Now.
  • Open Console Output to see the message.

Example 2: First Declarative Pipeline (without Jenkinsfile)

Goal: Create a pipeline with 3 stages.

  • New Item ‚Üí name: simple-pipeline ‚Üí choose Pipeline.
  • In the Pipeline script box, paste:
pipeline {
  agent any

  stages {
    stage('Build') {
      steps {
        echo 'Building...'
      }
    }

    stage('Test') {
      steps {
        echo 'Testing...'
      }
    }

    stage('Deploy') {
      steps {
        echo 'Deploying...'
      }
    }
  }
}

Save ‚Üí Build Now ‚Üí watch stages in the UI.

Example 3: Pipeline Using Jenkinsfile in Git Repo

In your Git repo (GitHub/GitLab), create Jenkinsfile:

pipeline {
  agent any

  stages {
    stage('Checkout') {
      steps {
        checkout scm
      }
    }

    stage('Build') {
      steps {
        echo 'Simulating build step'
      }
    }

    stage('Test') {
      steps {
        echo 'Simulating tests'
      }
    }
  }
}

In Jenkins:

  • New Item ‚Üí my-code-pipeline ‚Üí Pipeline.
  • Under Pipeline:
    • Definition: Pipeline script from SCM
    • SCM: Git
    • Repo URL: your repo URL
  • Save ‚Üí Build ‚Üí Jenkins pulls repo & uses Jenkinsfile.

Example 4: Simple Python Test Pipeline with Docker

Repo contains:

  • app.py
  • test_app.py

Create Jenkinsfile:

pipeline {
  agent {
    docker {
      image 'python:3.12-slim'
    }
  }

  stages {
    stage('Checkout') {
      steps {
        checkout scm
      }
    }

    stage('Install dependencies') {
      steps {
        sh 'pip install pytest'
      }
    }

    stage('Run tests') {
      steps {
        sh 'pytest -q'
      }
    }
  }
}

Jenkins will run the pipeline inside a Python Docker container and execute tests.

Example 5: Scheduled Job (Cron-style)

Goal: Run a job every night at 1 AM.

  • Open job configuration.
  • Under Build Triggers, enable Build periodically.
  • Set schedule: 0 1 * * *

This uses cron format: minute = 0, hour = 1, every day/month/weekday.
