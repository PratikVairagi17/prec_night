1. What is YAML?

YAML = ‚ÄúYAML Ain‚Äôt Markup Language‚Äù ‚Äì it‚Äôs a human-friendly format for configuration and data.

YAML is a human-readable data serialization format used to write configuration files and structured data. It is conceptually similar to JSON, but:

  • More readable for humans
  • Uses indentation instead of { } and [ ]
  • Very popular for configuration in DevOps, CI/CD, Kubernetes, Docker, etc.
JSON Example
{
"name": "pratik",
"role": "devops learner",
"skills": ["git", "docker", "linux"],
"experience": {
"years": 2,
"domain": "IT support"
}
}
YAML Example (same data)
name: pratik
role: devops learner
skills:
- git
- docker
- linux
experience:
years: 2
domain: IT support

Key Idea

YAML is easier for humans to read and write ‚Äì especially in big configuration files.

4. YAML Basics & Syntax

If you understand this section well, you‚Äôll avoid 80% of YAML errors in real projects.

4.1 Indentation is Everything (Spaces, NOT Tabs)

YAML uses indentation to represent structure.

  • Use spaces only (no tabs).
  • Common: 2 spaces per level (sometimes 4).
# Good
server:
  host: localhost
  port: 8080

# Bad (tabs or wrong spaces) ‚Üí parser errors

4.2 Key‚ÄìValue Pairs (Maps)

Basic structure:

name: pratik
age: 24
active: true

This is similar to a dictionary/object.

4.3 Lists (Sequences)

List of simple values:

skills:
  - git
  - docker
  - linux

List of objects:

users:
  - name: pratik
    role: devops
  - name: rahul
    role: backend

4.4 Data Types

string_value: "hello"
another_string: hello

integer_value: 10
float_value: 3.14

boolean_true: true
boolean_false: false

null_value: null # or ~

password: "abc:123#test"

Use quotes when you have :, #, or special characters.

4.5 Comments

# This is a comment
app:
  name: demo-app # inline comment

Everything after # is ignored by the parser.

4.6 Multiline Strings

Literal block | (keeps newlines):

description: |
  This is line one.
  This is line two.
  Newlines are preserved.

Folded block > (joins lines):

message: >
  This is a long line
  that will be folded
  into a single line.

4.7 Nested Structures

services:
  backend:
    image: my-backend:latest
    replicas: 3
    ports:
      - 8080
      - 8081
  frontend:
    image: my-frontend:latest
    replicas: 2
    ports:
      - 80

Very similar to what you see in Kubernetes or Docker Compose.

4.8 Anchors & Aliases (Advanced but Useful)

defaults: &default-config
  retries: 3
  timeout: 30

service_a:
  <<: *default-config
  url: https://service-a.example.com

service_b:
  <<: *default-config
  url: https://service-b.example.com

&default-config defines an anchor.
*default-config reuses it. <<: merges the map.

6. Hands-On YAML Examples & Tutorials

Try these yourself to make YAML feel natural.

Example 1: Simple config.yaml

Create a file config.yaml:

app:
  name: my-devops-app
  version: 1.0.0
  debug: true

server:
  host: 0.0.0.0
  port: 8080

database:
  host: localhost
  port: 5432
  user: admin
  password: "secret123"

features:
  - login
  - signup
  - reports

Task: Change debug to false and add another feature.

Example 2: Read YAML in Python

Install PyYAML:

pip install pyyaml

Create read_yaml.py:

import yaml

with open("config.yaml") as f:
  config = yaml.safe_load(f)

print("App Name:", config["app"]["name"])
print("Server Port:", config["server"]["port"])
print("Features:", config["features"])

Run python read_yaml.py to see values from your YAML config.

Example 3: Simple Docker Compose (YAML)

Create docker-compose.yml:

version: "3.8"

services:
  web:
    image: nginx:alpine
    ports:
      - "8080:80"

Run docker compose up and open http://localhost:8080 ‚Äì you used YAML to define a service.

Example 4: Small GitLab CI YAML

Create .gitlab-ci.yml:

  - test

  stage: test
  image: python:3.12-slim

Push to GitLab; it will automatically run this pipeline using your YAML config.

Example 5: Mini Kubernetes Deployment

Create deployment.yaml:

  name: nginx-deployment

  replicas: 2
  selector:
    matchLabels:

            - containerPort: 80

If you have kubectl configured:

kubectl apply -f deployment.yaml
kubectl get pods

Example 6: Indentation Practice ‚Äì Spot the Error

Original:

app:
  name: demo
version: 1.0 # Is this nested or top-level?

Here, version is top-level, not under app.

Correct version (if you want it under app):

app:
  name: demo
  version: 1.0

7. YAML vs JSON vs XML

Same idea (structured data), different trade-offs.

YAML

Best for:

  • DevOps configs (K8s, CI/CD)
  • Human-edited configuration

Pros:

  • Very readable
  • Supports comments
  • Minimal syntax

Cons:

  • Indentation errors
  • Weaker schema validation by default

JSON

Best for:

  • APIs, web apps, microservices
  • Data exchange backend ‚áÑ frontend

Pros:

  • Lightweight & ubiquitous
  • Easy to parse in any language

Cons:

  • No comments
  • Harder to read in large configs

XML

Best for:

  • Enterprise & legacy systems
  • Document-style data, SOAP, metadata

Pros:

  • Rich schema validation (XSD)
  • Handles complex structures & attributes

Cons:

  • Very verbose
  • Hard to write and read manually

Short Summary

  • Use YAML when humans will edit configs (K8s, CI/CD, Ansible).
  • Use JSON for APIs and data exchange between services.
  • Use XML when you need strong schema validation or work with older enterprise systems.

Bonus: YAML is essentially a superset of JSON ‚Äì JSON can be converted to YAML easily.
