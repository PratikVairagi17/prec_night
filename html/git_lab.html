1. What is GitLab?

Think of Git as the engine and GitLab as a full car with dashboard, controls, and automation.

GitLab is a web-based DevOps platform built around Git repositories. It provides:

  • Source code management (Git hosting)
  • Built-in CI/CD (GitLab CI)
  • Issue tracking & Kanban boards
  • Merge Requests (code review)
  • Container & package registry
  • Security & compliance (DevSecOps)

Git vs GitLab

Git: Distributed version control tool (CLI on your machine).

GitLab: Web platform built on top of Git with UI, CI/CD, issues, and more.

How You Can Use GitLab

GitLab.com (SaaS) Hosted by GitLab in the cloud.

Self-managed GitLab Install GitLab on your own server / VM (on-premise or cloud) for full control.

4. Important GitLab Concepts & Features

4.1 Project & Repository

A Project in GitLab includes:

  • Git repository
  • Issues & boards
  • Wiki, CI/CD settings
  • Registry, releases, environments

It‚Äôs the main unit where you push code and define pipelines.

4.2 Groups & Subgroups

A Group can contain multiple projects and subgroups.

Useful for:

  • Organising by team, product, or customer.
  • Managing permissions and policies centrally.

Example: group company-backend with projects auth-service, payment-service, etc.

4.3 Branches & Merge Requests (MRs)

Branches: main, dev, feature/login-api, etc.

MR (Merge Request): GitLab‚Äôs name for Pull Request.

Typical flow:

  • Create branch ‚Üí commit changes ‚Üí push.
  • Open MR to main.
  • CI pipeline runs; reviewers check code + tests.
  • After approval, MR is merged.

4.4 GitLab CI/CD Basics

Controlled by .gitlab-ci.yml in repo root.

  • Pipeline ‚Äì full CI run per push/MR.
  • Stages ‚Äì build, test, deploy.
  • Jobs ‚Äì tasks that run in stages.
  • Runners ‚Äì agents that execute jobs.

Jobs in the same stage run in parallel; stages run in order.

4.5 GitLab Runner

A Runner is a separate agent that pulls jobs from GitLab and executes them.

  • Installed on a VM, laptop, or Kubernetes cluster.
  • Types: shell, Docker, Kubernetes executors.
  • Can be shared (many projects) or specific (one project/group).

4.6 Container & Package Registry

Container Registry: Store Docker images like:

registry.gitlab.com/group/project/image:tag

Package Registry: Host private npm, Maven, PyPI, etc.

Useful for internal images and packages.

4.7 Issues, Boards & Milestones

  • Issues ‚Äì tasks, bugs, feature requests.
  • Boards ‚Äì Kanban: To Do ‚Üí Doing ‚Üí Done.
  • Milestones ‚Äì group issues for a release/sprint.

This allows you to manage project planning and tracking inside GitLab itself.

4.8 Protected Branches & Approvals (Bonus for Interviews)

  • Protect branches like main / production.
  • Only certain roles can push or merge to them.
  • Require approvals and passing pipelines before merge.

5. GitLab Interview Questions & Answers

Use these as quick revision before a DevOps / GitLab round.

GitLab is a web-based DevOps platform built around Git repositories. It provides source code management, built-in CI/CD pipelines, issue tracking, registries, and security features, allowing teams to manage the entire software development lifecycle in a single tool.
  • Git is a distributed version control system you install locally to track code history and collaborate using branches and commits.
  • GitLab is a platform that hosts Git repositories and adds a web UI, CI/CD, issue tracking, merge requests, registries, and more.
Both host Git repos and support collaboration. GitLab‚Äôs key differences:
  • GitLab has tightly integrated, built-in CI/CD (GitLab CI) using .gitlab-ci.yml.
  • GitLab is widely used for self-hosted, enterprise DevOps platforms.
  • GitHub is extremely popular for open-source; GitLab also supports open source but positions itself as a ‚Äúcomplete DevOps platform‚Äù.
GitLab CI/CD is the built-in continuous integration and delivery/deployment system in GitLab. Pipelines are defined in .gitlab-ci.yml, and GitLab automatically runs jobs on runners when code is pushed, merge requests are created, or schedules are triggered.
.gitlab-ci.yml is a YAML configuration file stored in the root of your repository. It defines:
  • Pipeline stages (e.g., build, test, deploy)
  • Jobs and the scripts they run
  • Images, artifacts, caches, variables
  • Rules and conditions for running jobs
GitLab reads this file to know which jobs to run and in what order.
A GitLab Runner is an agent/service that executes CI/CD jobs for GitLab. It connects to a GitLab instance, picks up jobs defined in .gitlab-ci.yml, runs them (via shell, Docker, or Kubernetes), and sends logs and status back to GitLab.
  • Shared runner: Available to multiple projects or the whole instance. Good for general workloads shared by many teams.
  • Specific runner: Assigned to a particular project or group. Used when jobs need special tools, higher isolation, or dedicated capacity.
  • Stages are logical steps in a pipeline, such as build, test, deploy. Stages run in sequence.
  • Jobs are individual tasks within stages. All jobs in a stage run in parallel by default. The pipeline moves to the next stage only when all jobs in the current stage succeed (unless configured otherwise).
stages:
  - build
  - test

build_job:
  stage: build
  script:
    - echo "Building the app"

test_job:
  stage: test
  script:
    - echo "Running tests"
This defines two stages (build and test) and one job per stage.
Artifacts are files produced by a job (e.g., build outputs, test reports, coverage reports) that GitLab stores after the job finishes. They can be:
  • Downloaded from the pipeline UI.
  • Used by later jobs in the pipeline.
Example: 
build_job:
  stage: build
  script:
    - make build
  artifacts:
    paths:
      - build/
Environments represent where your application runs (e.g., dev, staging, production). Deployment jobs in GitLab CI target a specific environment. GitLab tracks:
  • Last deployment per environment
  • Who deployed and when
  • History and status of environments
Variables can be defined:
  • In .gitlab-ci.yml under variables:
  • In project/group settings ‚Üí CI/CD ‚Üí Variables (useful for secrets)
Example: 
variables:
  APP_ENV: "production"

deploy:
  stage: deploy
  script:
    - echo "Deploying to $APP_ENV"
  • Store secrets as CI/CD variables in project/group settings (masked & protected).
  • Use them in jobs as environment variables, e.g., $DB_PASSWORD.
  • Never hard-code secrets in .gitlab-ci.yml or source code.
  • Optionally integrate with external secret managers like HashiCorp Vault or cloud KMS.
Yes. GitLab has a self-managed edition that can be installed on company servers or cloud VMs. Reasons to self-host:
  • Keep source code and pipelines inside the private network.
  • Meet regulatory and compliance requirements.
  • Control upgrades, configuration, and integrations.
  • Use internal authentication (LDAP/AD, SSO).
Typical workflow:
  • Developer pushes code to a feature branch ‚Üí pipeline runs (build + tests).
  • Developer opens a Merge Request to main ‚Üí pipeline runs again.
  • Reviewers check the MR, code, and pipeline results; approvals are given.
  • After merge to main, another pipeline runs with a deploy job that deploys to staging or production.

6. Hands-On GitLab Examples & Tutorials

You can try all of this on GitLab.com (free) with your friends.

Example 1: Create a Project & Push Code

Step 1 ‚Äì Create a project on GitLab

  • Log in to GitLab.
  • Click New project ‚Üí Create blank project.
  • Name it gitlab-demo, choose visibility, and create.
  • Copy the HTTPS URL, e.g. https://gitlab.com/your-username/gitlab-demo.git

Step 2 ‚Äì Clone and push code

git clone https://gitlab.com/your-username/gitlab-demo.git
cd gitlab-demo

# app.py
def add(a, b):
    return a + b

if __name__ == "__main__":
    print(add(2, 3))

git add app.py
git commit -m "Add simple add function"
git push origin main

Now your code is stored in GitLab.

Example 2: Basic GitLab CI Pipeline (.gitlab-ci.yml)

Create .gitlab-ci.yml in the project root:

stages:
  - test

test_python:
  stage: test
  image: python:3.12-slim
  script:
    - python --version
    - python app.py

Commit and push this file. GitLab will detect .gitlab-ci.yml, start a pipeline, run a Python container, and execute your script. Check it under CI/CD ‚Üí Pipelines.

Example 3: Pytest Integration (Real CI)

Step 1 ‚Äì Add a test (test_app.py):

from app import add

def test_add():
    assert add(2, 3) == 5

Step 2 ‚Äì Update .gitlab-ci.yml:

stages:
  - test

pytest_job:
  stage: test
  image: python:3.12-slim
  script:
    - pip install pytest
    - pytest

Push your changes. Now every push runs pytest automatically ‚Äì this is real Continuous Integration.

Example 4: Multi-stage Pipeline with Artifacts

stages:
  - build
  - test

build_job:
  stage: build
  image: python:3.12-slim
  script:
    - echo "Simulating build step"
    - mkdir build
    - echo "Build artifact" > build/output.txt
  artifacts:
    paths:
      - build/

test_job:
  stage: test
  image: python:3.12-slim
  dependencies:
    - build_job
  script:
    - ls -R
    - cat build/output.txt
    - echo "Running tests here"

build_job creates an artifact in build/. test_job depends on it and reads that file ‚Äì this shows how artifacts move between stages.

Example 5: Basic Docker Build in GitLab CI (Concept)

For a project with Container Registry enabled and a Docker executor runner:

stages:
  - build

docker_build:
  stage: build
  image: docker:latest
  services:
    - docker:dind
  variables:
    DOCKER_DRIVER: overlay2
  script:
    - docker login -u "$CI_REGISTRY_USER" -p "$CI_REGISTRY_PASSWORD" "$CI_REGISTRY"
    - docker build -t "$CI_REGISTRY_IMAGE:latest" .
    - docker push "$CI_REGISTRY_IMAGE:latest"

This builds a Docker image and pushes it to GitLab‚Äôs Container Registry for that project.
