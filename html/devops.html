What is DevOps?

DevOps = Development (Dev) + Operations (Ops) working together using automation, tools, and processes to deliver software faster, more reliably, and continuously.

Traditional Approach (Siloed)

Developers: Write code, throw it over the wall

Operations: Deploy, maintain, handle crashes

Result: Silos, delays, slow releases

DevOps Approach (Unified)

‚úì One combined or tightly aligned team

‚úì Shared responsibility throughout the pipeline

‚úì Heavy automation: CI/CD, IaC, monitoring

‚úì Small, frequent changes instead of big releases

Key Principles

Collaboration: Dev, Ops, QA, Security work as one team

Automation: CI/CD, Infrastructure as Code

Continuous Integration: Frequent code pushes and auto tests

Continuous Delivery/Deployment: Always deployable

Infrastructure as Code: Version control for infrastructure

Monitoring: Logs, metrics, continuous improvement

DevOps Lifecycle & Tools

1. Plan

Tools: Jira, Trello, GitHub Projects

Define requirements and create tasks

2. Code

Tools: Git, GitHub, GitLab

Branches, PRs, code reviews

3. Build

Tools: Maven, Gradle, npm, Docker

4. Test

Tools: JUnit, PyTest, Selenium

Interview Questions & Answers

DevOps is a culture and set of practices that bring development and operations together, using automation, CI/CD, and collaboration to deliver software faster, more reliably, and continuously.
Agile focuses on the development process. DevOps focuses on build, test, deploy, and operations. They complement each other: Agile on process, DevOps on delivery and operations.
CI (Continuous Integration): Developers merge code frequently. Each merge triggers automated build and tests. CD (Continuous Delivery/Deployment): Code is always ready for deployment after tests pass, or auto-deployed to production.
IaC means managing infrastructure (servers, networks, databases) with code files instead of manual steps. Tools: Terraform, CloudFormation, Ansible. Allows versioning, repeatability, and automation.
Continuous Delivery: System always deployable, but manual approval needed for production. Continuous Deployment: Every change that passes tests is automatically deployed to production with no manual gate.
Container: Lightweight, shares OS kernel, starts in seconds (Docker). VM: Full OS on hypervisor, heavier, more resource usage, slower to start.
Docker is a platform that packages application and dependencies into a container image that runs consistently across all environments.
Kubernetes is a container orchestration platform that automates deployment, scaling, and management of containerized applications across multiple machines.
Two production environments: Blue (current) and Green (new). Deploy to Green, test it, then switch traffic from Blue to Green. Quick rollback if issues occur.
Deploy new version to a small percentage of users first (e.g., 5%). If everything looks good, gradually increase traffic to 100%.
Managing system configurations in a consistent state (packages, services, files, users). Tools: Ansible, Puppet, Chef, SaltStack.
Ability to understand system internal state based on logs, metrics, and traces. Makes debugging easier and improves performance.
Git is the single source of truth for application and infrastructure configurations. A controller continuously reconciles the actual environment with Git definitions.
Never hardcode secrets. Use tools like AWS Secrets Manager, HashiCorp Vault, or Kubernetes Secrets (encrypted). Limit access via IAM/roles.

Hands-On Examples

Example 1: GitHub Actions CI

name: Python CI
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: python -m pytest

Every push or pull request triggers this workflow and runs your Python tests automatically.

Example 2: Simple Dockerfile

FROM python:3.11-slim
WORKDIR /app
COPY . /app
RUN pip install -r requirements.txt
CMD ["python", "app.py"]

Build with docker build -t myapp . and run with docker run -p 8000:8000 myapp.
