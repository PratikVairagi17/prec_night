State Management

What is State?

The state file (terraform.tfstate) tracks all resources created by Terraform. It's the source of truth.

State File Contents:

  • Resource IDs and configurations
  • Outputs and variables
  • Resource dependencies
  • Metadata and timestamps
Important: Never edit state files manually or commit to Git!

State Commands:

terraform state list # List all resources terraform state show resource # Show resource details terraform state rm resource # Remove from state terraform state pull # Get current state terraform state push # Update state

Variables & Outputs

Variables (Input)

variable "instance_type" { description = "EC2 type" type = string default = "t2.micro" }

Outputs (Return Values)

output "instance_id" { description = "EC2 ID" value = aws_instance.web.id }

Variable Types:

string # Text number # Integer or float bool # true/false list() # [1, 2, 3] map() # {key = value} object() # Complex structures

Data Sources

Data sources fetch information from existing resources without creating new ones.

Example: Get Latest AMI

data "aws_ami" "ubuntu" { most_recent = true filter { name = "name" values = ["ubuntu/images/hvm-ssd/ubuntu-*"] } } resource "aws_instance" "web" { ami = data.aws_ami.ubuntu.id }

Advanced Terraform Concepts

1. Dynamic Blocks

Generate repeated configuration blocks dynamically:

resource "aws_security_group" "web" { dynamic "ingress" { for_each = var.ingress_rules content { from_port = ingress.value.from_port to_port = ingress.value.to_port protocol = ingress.value.protocol cidr_blocks = ingress.value.cidr_blocks } } }

2. Count vs For_Each

Count (Numeric)

resource "aws_instance" "web" { count = 3 instance_type = "t2.micro" } output "ids" { value = aws_instance.web[*].id }

For_Each (Map)

resource "aws_instance" "web" { for_each = var.instances instance_type = each.value.type } output "ids" { value = [for k, v in aws_instance.web : v.id] }

3. Locals

Define local values for repeated configurations:

locals { common_tags = { Environment = var.env Project = "MyApp" Terraform = true } instance_count = var.env == "prod" ? 5 : 1 } resource "aws_instance" "web" { count = local.instance_count tags = local.common_tags }

4. Conditional Logic

variable "create_db" { type = bool } resource "aws_db_instance" "mysql" { count = var.create_db ? 1 : 0 allocated_storage = 20 }

5. Workspaces

Manage multiple environments with the same code:

terraform workspace new dev terraform workspace new prod terraform workspace select dev terraform workspace list

6. Remote Backends

terraform { backend "s3" { bucket = "my-tf-state" key = "prod/terraform.tfstate" region = "ap-south-1" encrypt = true dynamodb_table = "terraform-locks" } }
Remote backends enable team collaboration and state locking

Interview Questions - Basic Level (1-10)

Q1: What is Terraform?

Terraform is an open-source Infrastructure as Code tool by HashiCorp that allows you to define, deploy, and manage cloud infrastructure using declarative configuration files.

Q2: What is Infrastructure as Code?

IaC means defining IT infrastructure using code instead of manual configuration. Benefits: automation, version control, consistency, repeatability, and reduced human errors.

Q3: What is a provider?

A provider is a plugin connecting Terraform to a cloud platform (AWS, Azure, GCP, etc.). It contains all API logic for creating resources.

Q4: What is a resource?

A resource is an infrastructure component you create (EC2 instance, S3 bucket, VPC, security group, etc.).

Q5: What does terraform init do?

Initializes a Terraform project by downloading provider plugins, setting up backend, and preparing the environment. Must be run before plan/apply.

Q6: What does terraform plan do?

Shows a preview of changes Terraform will make without making actual changes. Displays what will be created, changed, or destroyed.

Q7: What does terraform apply do?

Executes the plan and creates/updates actual infrastructure in the cloud. Asks for confirmation before proceeding.

Q8: What does terraform destroy do?

Removes all resources created by Terraform. Used for cleaning up and avoiding unnecessary cloud bills.

Q9: What is terraform validate?

Validates the configuration syntax and internal logic. Checks for missing arguments, incorrect types, and invalid HCL.

Q10: Where does Terraform store infrastructure info?

In the state file (terraform.tfstate). It stores resource IDs, configurations, and dependencies - it's Terraform's source of truth.

Interview Questions - Intermediate Level (11-20)

Q11: What is the .terraform directory?

Contains provider plugins, module downloads, and backend data. Created automatically during terraform init.

Q12: Explain terraform.tfstate purpose

Stores current infrastructure status. Terraform compares desired state (from .tf files) with current state (tfstate) to determine changes.

Q13: Why shouldn't state files be edited manually?

Manual editing corrupts Terraform's understanding of resources, causes mismatches, and breaks apply/plan/destroy commands. Only Terraform should manage state.

Q14: Difference between variables.tf and terraform.tfvars

variables.tf defines variables (name, type, description). terraform.tfvars assigns real values to those variables.

Q15: What are output variables?

Display useful information after Terraform applies resources (instance IDs, public IPs, DB endpoints, etc.).

Q16: What are data sources?

Allow Terraform to fetch data from existing resources without creating new ones. Example: getting latest AMI ID.

Q17: How is Terraform integrated with Git?

Store .tf files in Git for version control. Use .gitignore for sensitive files. Enable CI/CD pipelines for automated terraform plan/apply.

Q18: How does Terraform handle dependencies?

Automatically builds dependency graphs using resource references and depends_on arguments. Ensures correct creation order.

Q19: What is resource drift?

Occurs when infrastructure is changed manually outside Terraform. Detected in terraform plan as differences between desired and actual state.

Q20: terraform import vs terraform state

terraform import brings existing cloud resources into Terraform management. terraform state inspects/modifies the state file.

Interview Questions - Advanced Level (21-30)

Q21: What are backends in Terraform?

Determine where state file is stored. Types: Local (default), S3, Azure storage, Consul. Used in teams for sharing state, locking, and backup.

Q22: How does remote state management work?

State file stored in S3. Locking via DynamoDB. Multiple users share same state. Prevents conflicts and enables team collaboration.

Q23: Terraform dependency management

Builds dependency graph based on references and depends_on arguments. Ensures resources created in correct order.

Q24: local-exec vs remote-exec

local-exec runs on local machine/CI server. remote-exec runs on remote machine (EC2 instance).

Q25: What is lifecycle meta-argument?

Controls how Terraform handles resources. Options: create_before_destroy, ignore_changes, prevent_destroy.

Q26: How does Terraform handle resource drift?

Detects drift via terraform plan. Compares config vs real infrastructure. Updates/recreates to match desired state.

Q27: What are Terraform workspaces?

Allow multiple environments using same config (dev, test, prod). Created with: terraform workspace new dev.

Q28: How do modules work?

Reusable Terraform code packages. Create modules for VPC, EC2, RDS, etc. Promotes DRY principle and maintainability.

Q29: How to manage secrets?

Use AWS SSM Parameter Store, Secrets Manager, HashiCorp Vault, environment variables, or encrypted backend. Never store in tfvars.

Q30: What are Terraform best practices?

Use modules, separate environments, remote backend, workspaces, .gitignore, variables/outputs, terraform fmt, validate with CI, keep modular.

Best Practices for Large Projects

  • Use modules - Create reusable templates for VPC, EC2, RDS, etc.
  • Separate environments - Keep dev, test, and prod configurations distinct
  • Remote backend - Store state in S3 with locking via DynamoDB
  • Use workspaces - Manage multiple environments efficiently
  • .gitignore - Never commit state files, credentials, or sensitive data
  • Variables & outputs - Make code flexible and reusable
  • terraform fmt - Format code consistently
  • CI/CD validation - Automate plan/apply in pipelines
  • Keep modular - Small, focused modules are easier to maintain
  • Document code - Add comments explaining complex logic

Security Best Practices

Never commit these to Git:
  • terraform.tfstate files
  • AWS access keys
  • Passwords or secrets
  • Private certificates
  • API tokens

Secret Management Solutions:

  • AWS Secrets Manager - Store database passwords, API keys
  • AWS SSM Parameter Store - Store configuration values
  • HashiCorp Vault - Centralized secrets management
  • Environment variables - Pass secrets at runtime
  • Encrypted backend - Use encrypted S3 backend

Encryption:

  • Enable S3 encryption for state files
  • Use KMS for encryption keys
  • Enable DynamoDB encryption for state locking
  • Use encrypted connections (TLS/HTTPS)
