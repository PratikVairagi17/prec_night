1. What is Git?

Git is a Distributed Version Control System (DVCS) that helps developers track changes in code, collaborate with others, and revert to previous versions easily. Every developer has a complete copy of the project history on their local machine.

Key Features:

  • Tracks changes in source code with detailed history
  • Enables teamwork via branches and merging
  • Distributed - every user has a full copy of the project
  • Offline capability - works without internet, syncs when online
  • Commit history - every version can be rolled back or compared

2. What is GitHub?

GitHub is a cloud-based hosting platform for Git repositories. It helps you store, share, and collaborate on Git projects online with teams around the world.

Key Features:

  • Remote repository hosting - store projects in the cloud
  • Collaboration via Pull Requests (PRs) and code reviews
  • Access control - manage who can view/edit code
  • CI/CD integration - automated testing and deployment
  • Integration with tools - VS Code, Docker, Jenkins, etc.
üí° Think of it like:
‚Ä¢ Git = Local tool for version control on your machine
‚Ä¢ GitHub = Online home for your Git projects (remote storage)

5. Branching in Git

A branch is a separate line of development used for adding new features or fixing bugs without disturbing the main code. Branches allow multiple developers to work independently.

Branch Commands:

git branch feature1 # Create a new branch git checkout feature1 # Switch to that branch git checkout -b bugfix # Create & switch in one command git branch # List all branches git branch -a # List all branches (local & remote) git merge feature1 # Merge feature1 into current branch git branch -d feature1 # Delete the branch

Typical Branch Workflow:

  1. Create a new branch for your feature
  2. Switch to that branch
  3. Make changes and commit
  4. Push branch to GitHub
  5. Create Pull Request for review
  6. Merge after approval
  7. Delete the branch

6. Git Merge - Detailed Explanation

Merging combines changes from one branch into another. It preserves all commit history and is the standard way to integrate features in Git.

Basic Merge Process:

git checkout main # Switch to main branch git merge feature1 # Merge feature1 into main

Types of Merges:

1. Fast-Forward Merge

Occurs when the main branch hasn't changed since the feature branch was created. Git simply moves the branch pointer forward.

Before: main ‚Üí A ‚Üí B feature1 ‚Üí B After: main ‚Üí A ‚Üí B (pointer moved forward)

2. Three-Way Merge

Occurs when both branches have new commits. Git combines changes and creates a new "merge commit".

Before: main ‚Üí A ‚Üí B ‚Üí D feature1 ‚Üí A ‚Üí C After: main ‚Üí A ‚Üí B ‚Üí D ‚Üí M (merge commit) ‚îî‚Üí C ‚îÄ‚îò

Merge Conflicts:

‚ö†Ô∏è Conflict Situation: Occurs when both branches modified the same lines in a file. Git cannot automatically decide which change to keep.

Resolving Conflicts:

# 1. Git marks conflicts with: <<<<<<< HEAD your changes ======= their changes >>>>>>> branch-name # 2. Manually edit the file to keep desired code # 3. Stage and commit git add <file> git commit -m "Resolved merge conflict"

7. Git Reset - Detailed Explanation

Git reset is used to undo commits or unstage files by moving the HEAD pointer to a specific previous commit. It can be dangerous if used incorrectly.

Types of Reset:

Reset Type Keeps Changes? Keeps Staged? Use Case
--soft ‚úÖ Yes ‚úÖ Yes Undo commit but keep changes staged
--mixed (default) ‚úÖ Yes ‚ùå No Unstage changes but keep files modified
--hard ‚ùå No ‚ùå No Completely remove commits and changes

Soft Reset Example:

git reset --soft HEAD~1

‚úÖ Removes the last commit but keeps your changes staged, so you can recommit with a different message.

Mixed Reset Example:

git reset --mixed HEAD~1

‚úÖ Removes the last commit and unstages changes, but files remain modified. You'll need to git add again.

Hard Reset Example:

git reset --hard HEAD~1

‚ùå Completely removes the last commit and all changes. This is dangerous and hard to undo!

‚ö†Ô∏è Warning: git reset --hard deletes data. Use it only when you're absolutely sure. Consider using git revert as a safer alternative for shared repositories.

8. Merge vs Reset - Key Differences

These are two fundamental Git operations that serve different purposes and have different implications for your repository.

Feature git merge git reset
Purpose Combine two branches together Undo commits or move HEAD pointer
Effect Creates new commit (merge commit) Moves HEAD to previous commit
History Preserved completely Rewritten (danger of losing commits)
Collaboration Safe for shared branches Should only be used locally
Risk Level Low - no data loss High (especially --hard)
Typical Usage Integrating new features into main Undoing mistakes before pushing

When to Use MERGE:

  • You want to combine work from another branch into current branch
  • Working in a collaborative/team environment
  • Merging feature branch into main/production
  • You need complete audit trail of all changes
  • Safe option - preserves all history

When to Use RESET:

  • You want to undo local commits that haven't been pushed
  • You need to move HEAD to a previous commit
  • You're working alone and want to clean up history
  • You made a mistake and want to go back
  • Should NOT use on commits already pushed to shared repo

Real-World Scenario:

Scenario 1: Feature Integration
You've completed work on feature-branch and want to add it to main.
‚ûú Use MERGE - combines branches safely

Scenario 2: Undo Local Mistake
You committed wrong code locally but haven't pushed yet.
‚ûú Use RESET - undo local commits safely

Scenario 3: Already Pushed to Shared Repo
You pushed wrong code to GitHub and team sees it.
‚ûú Use REVERT - creates new commit that undoes changes (safer than reset)

9. Git Rebase - Detailed Explanation

Rebase is an alternative to merge that rewrites commit history to create a linear sequence of commits. It "replays" commits from one branch onto another branch.

What is Rebase?

Rebase takes commits from one branch and applies them on top of another branch, creating a cleaner, linear history compared to merge which creates a branching history.

Basic Rebase Syntax:

git rebase <branch-name>

Visual Comparison: Merge vs Rebase

Before (Both branches have new commits):

main: A -- B -- D / feature: A -- C

After Using MERGE:

main: A -- B -- D -- M (merge commit) / / feature: A -- C ---

Result: Non-linear history with merge commit

After Using REBASE:

main: A -- B -- D feature: A -- B -- D -- C'

Result: Linear history, commits replayed cleanly

Rebase Workflow Example:

# 1. Switch to feature branch git checkout feature-branch # 2. Rebase onto main git rebase main # 3. Resolve any conflicts if they occur # Edit conflicting files, then: git add . git rebase --continue # 4. Switch back to main git checkout main # 5. Fast-forward merge (now just moves pointer) git merge feature-branch

Types of Rebase:

1. Simple Rebase (Linear History)

git rebase main

Replays all commits from feature branch on top of main branch.

2. Interactive Rebase (Edit commits)

git rebase -i HEAD~3

Edit last 3 commits: squash, reorder, edit messages, or delete commits.

3. Interactive Rebase Options:

  • pick = Use commit as is
  • reword = Use commit but edit message
  • squash = Combine with previous commit
  • fixup = Combine and discard message
  • drop = Remove commit entirely

Rebase Conflicts:

Similar to merge, rebase can have conflicts. Resolve the same way:

# 1. Git stops at conflicting commit # 2. Edit files and resolve conflicts # 3. Stage resolved files git add . # 4. Continue rebase git rebase --continue # 5. Or abort if you want to cancel git rebase --abort
‚ö†Ô∏è Important Rules About Rebase:
‚Ä¢ NEVER rebase public/shared commits - use only on local branches
‚Ä¢ NEVER rebase commits already pushed to GitHub
‚Ä¢ Use merge for shared branches (main, develop)
‚Ä¢ Use rebase for local feature branches before pushing
‚Ä¢ Rebase rewrites history - dangerous on shared repos!

11. Git Delete Operations

Delete a Branch (Local):

git branch -d feature1 # Delete after merge (safe) git branch -D feature1 # Force delete (even if not merged)

Delete a Branch (Remote - GitHub):

git push origin --delete feature1 # Delete from GitHub

Delete a File and Track Deletion:

git rm filename.txt git commit -m "Deleted filename.txt"

Untrack File But Keep Locally:

git rm --cached filename.txt git commit -m "Untracked filename.txt"

This removes the file from Git tracking but keeps it on your machine (useful for config files, secrets, etc.)

13. Interview Questions & Answers

Q1. What is Git and why is it important?

A: Git is a distributed version control system that tracks changes in source code. It's important because it:
‚Ä¢ Enables team collaboration on code
‚Ä¢ Maintains complete history of all changes
‚Ä¢ Allows rollback to previous versions
‚Ä¢ Prevents data loss with multiple copies
‚Ä¢ Supports parallel development through branching
Q2. What is the difference between Git and GitHub?

A:
‚Ä¢ Git = A version control tool that runs locally on your computer
‚Ä¢ GitHub = A cloud platform that hosts Git repositories for collaboration
Q3. What is a branch and why do we use branches?

A: A branch is a separate line of development. We use branches to:
‚Ä¢ Work on new features without affecting main code
‚Ä¢ Allow multiple developers to work independently
‚Ä¢ Test code before merging into production
‚Ä¢ Organize work by feature or bugfix
Q3b. What is the difference between merge, rebase, and reset?

A:
‚Ä¢ Merge = Combines branches, creates merge commit, preserves history
‚Ä¢ Rebase = Replays commits linearly, rewrites history, cleaner but risky
‚Ä¢ Reset = Moves HEAD pointer, undoes commits, only for local use
Q4. What is the difference between merge and reset?

A:
‚Ä¢ Merge = Combines changes from two branches, creates new commit, preserves history
‚Ä¢ Reset = Moves HEAD pointer to previous commit, can rewrite history, used for undoing
Q5. When should you use merge vs reset?

A:
‚Ä¢ Use MERGE when: Combining feature branches into main, working with teams
‚Ä¢ Use RESET when: Undoing local commits not yet pushed, cleaning up local history
Q6. What are the types of git reset?

A:
‚Ä¢ --soft = Undo commit, keep changes staged (ready to recommit)
‚Ä¢ --mixed = Undo commit, keep changes unstaged (need to git add again)
‚Ä¢ --hard = Undo commit and delete all changes (dangerous!)
Q7. What happens during a merge conflict?

A: A merge conflict occurs when two branches modified the same lines in a file differently. Git cannot automatically decide which version to keep. You must:
‚Ä¢ Manually edit the file to resolve conflict
‚Ä¢ Remove conflict markers (<<<, ===, >>>)
‚Ä¢ Stage the resolved file
‚Ä¢ Commit the merge
Q8. What is a Pull Request (PR)?

A: A Pull Request is a request to merge changes from one branch (usually feature branch) into another (like main). It allows team members to:
‚Ä¢ Review code before merging
‚Ä¢ Discuss changes and suggestions
‚Ä¢ Run automated tests
‚Ä¢ Maintain code quality
Q9. How do you undo the last commit?

A: There are two common ways:

Option 1 (Rewrite History - Local Only):
git reset --soft HEAD~1 - keeps changes staged
git reset --hard HEAD~1 - deletes changes

Option 2 (Safe - Creates New Commit):
git revert HEAD - creates new commit that undoes the change
Q10. What is the difference between fetch and pull?

A:
‚Ä¢ git fetch = Downloads changes from remote (doesn't merge)
‚Ä¢ git pull = Downloads AND merges changes from remote (fetch + merge)
Q11. What should you do before pushing to GitHub?

A:
‚Ä¢ Run tests to ensure code works
‚Ä¢ Check for conflicts by pulling latest changes
‚Ä¢ Review your own code
‚Ä¢ Write meaningful commit messages
‚Ä¢ Verify you're on the correct branch
Q12. How do you recover a deleted branch?

A: If you accidentally deleted a branch, you can recover it using:
git reflog - shows all branch operations
git checkout -b branch-name SHA-ID - recreate branch from SHA
Q13. What is rebase and how is it different from merge?

A: Rebase rewrites commit history to create linear sequence of commits. Key differences:
‚Ä¢ Merge: Non-linear history, creates merge commit, safe for shared branches
‚Ä¢ Rebase: Linear history, replays commits, cleaner but risky for public commits
Q14. When should you use rebase?

A: Use rebase ONLY when:
‚Ä¢ Working on LOCAL feature branches
‚Ä¢ Commits haven't been pushed to GitHub yet
‚Ä¢ You're the only one working on the branch
‚Ä¢ You want clean, linear commit history

NEVER rebase: Public commits, shared branches, or anything already pushed
Q15. What is interactive rebase?

A: Interactive rebase (git rebase -i) allows you to edit commits:
‚Ä¢ pick = Keep commit
‚Ä¢ squash = Combine with previous commit
‚Ä¢ reword = Change commit message
‚Ä¢ drop = Remove commit

Usage: git rebase -i HEAD~3 (edit last 3 commits)
Q16. What is the golden rule of rebasing?

A: "Never rebase public history"
If commits are already pushed to shared repository and others have pulled them, DO NOT rebase. Use merge or revert instead. Rebase rewrites history which can cause problems for team members.
